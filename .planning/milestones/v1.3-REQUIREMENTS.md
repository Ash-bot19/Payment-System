# Requirements: v1.3 ML Risk Scoring

## Milestone Scope

This requirements file covers **v1.3 ML Risk Scoring** — the XGBoost inference layer that scores validated payment events for fraud risk.

## Requirements

### v1.3 Requirements

- [x] XGBoost ML risk scoring service consuming `payment.transaction.validated` via Kafka
- [x] 8-feature input vector assembled from Redis `feat:{event_id}` hash (written by Spark)
- [x] Inference SLA: p99 < 100ms local; model loaded once at startup (load-once pattern)
- [x] Redis feature lookup with 3×50ms retry; 20ms socket timeout → immediate fallback to `FEATURE_DEFAULTS`
- [x] Fallback path: `FEATURE_DEFAULTS` + `manual_review=True` when features unavailable (Redis miss or timeout)
- [x] `SCORING → AUTHORIZED` (risk < 0.7) and `SCORING → FLAGGED` (risk ≥ 0.7) state transitions written to `payment_state_log`
- [x] DB write before Kafka publish (state persisted before scored event published downstream)
- [x] Publish `ScoredPaymentEvent` to `payment.transaction.scored` after each inference
- [x] Publish to `payment.alert.triggered` only when `is_high_risk=True` (risk ≥ 0.7)
- [x] Application-level idempotency guard: skip duplicate `SCORING` state writes via DB query
- [x] FastAPI `POST /score` debug endpoint on port 8001 accepting `FeatureVector` (8 fields), returning `RiskScore`
- [x] `model.ubj` (38KB, AUC=1.0) committed to repo — zero runtime dependency on training infrastructure
- [x] `scoring-consumer` and `ml-scoring-service` containerised with health checks in Docker Compose
- [x] 47 tests: 13 unit (scorer) + 34 unit (consumer + ml_service) + 4 E2E — all passing
- [x] Human UAT passed: both containers healthy, POST /score returned `risk_score=0.00336`, `xgboost_model_loaded` in logs
